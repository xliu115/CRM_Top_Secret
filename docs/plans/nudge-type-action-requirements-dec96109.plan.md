# Nudge Type Action Requirements

## Current State (Implemented)

The nudge system supports 9 nudge types with a **consolidated card model** — one nudge per contact, merging all relevant insights into a single actionable card. Key capabilities:

- **9 nudge types** fully supported: STALE_CONTACT, JOB_CHANGE, COMPANY_NEWS, UPCOMING_EVENT, MEETING_PREP, EVENT_ATTENDED, EVENT_REGISTERED, ARTICLE_READ, LINKEDIN_ACTIVITY
- **Consolidated nudge model**: The engine produces one nudge per contact, collecting all insights (stale, executive transition, news, events, LinkedIn, articles) into a `metadata.insights[]` array. This eliminates the noise of 20+ separate cards per contact.
- **Type-specific CTAs and context**: Each insight type has its own icon, color, label, and contextual rendering within the consolidated card
- **In-app LLM-powered email draft panel**: `DraftEmailPanel` generates personalized outreach via `POST /api/nudges/[id]/draft-email` with type-specific prompts and editable subject/body
- **Meeting brief generation**: `MeetingBriefPanel` generates structured briefs via `POST /api/nudges/[id]/meeting-brief` with attendee insights, agenda, questions, and preparation checklist
- **Partner relationship context**: JOB_CHANGE insights show which McK partners know the person who changed jobs
- **Push email digest**: HTML digest email with type-specific icons, colors, CTA labels, and a type breakdown summary

## Design Principles

1. **One card per contact** — All reasons to reach out are consolidated into a single card, reducing noise and decision fatigue
2. **Context first** — Each insight provides the "why now" with type-specific details, signal content, and source links
3. **One-click outreach** — Pre-drafted email or meeting brief generated by LLM, editable in-app before sending
4. **Priority-driven** — The highest-priority insight determines the card's primary CTA and sort order

## Nudge Consolidation Architecture

### Problem

The original design created one nudge per signal per contact. A single contact at a large company could generate 20+ nudge cards (1 stale + 1 executive transition + 15 company news + 2 LinkedIn activities + 1 article read). This created overwhelming noise and made the nudges page unusable.

### Solution

The engine now produces **one nudge per contact** with all insights merged into `metadata.insights[]`.

**Priority ranking** determines which insight type becomes the card's primary `ruleType`:

1. MEETING_PREP (highest)
2. STALE_CONTACT
3. JOB_CHANGE
4. LINKEDIN_ACTIVITY
5. EVENT_ATTENDED
6. EVENT_REGISTERED
7. ARTICLE_READ
8. UPCOMING_EVENT
9. COMPANY_NEWS (lowest)

**Signal caps** prevent any single type from dominating:

- Company news: max 3 per contact
- LinkedIn activity: max 2 per contact

**Reason text** is a consolidated summary:

- Single insight: uses the insight's own reason text
- Multiple insights: "5 reasons to reach out to Alex Thompson at Google (Alphabet): overdue for a check-in, executive transition, and company news."

**Key implementation files:**

- [src/lib/services/nudge-engine.ts](src/lib/services/nudge-engine.ts) — `Insight` interface, `pickPrimary()`, `buildReason()`, consolidated loop
- [src/app/nudges/page.tsx](src/app/nudges/page.tsx) — `InsightItem` component, consolidated `NudgeCard`

### Metadata Schema

Each nudge's `metadata` field contains a JSON object:

```json
{
  "insights": [
    {
      "type": "STALE_CONTACT",
      "reason": "No interaction in 43 days (threshold: 30 days).",
      "priority": "URGENT"
    },
    {
      "type": "JOB_CHANGE",
      "reason": "Executive transition at Google (Alphabet): \"Brian Foster announced new role: CTO\".",
      "priority": "MEDIUM",
      "signalId": "sig-0121",
      "signalContent": "Brian Foster announced new role: CTO at Google (Alphabet)",
      "signalUrl": "https://linkedin.com/...",
      "relatedPartners": [{"partnerId": "p-sam-rivera", "partnerName": "Sam Rivera"}],
      "personName": "Brian Foster"
    },
    {
      "type": "COMPANY_NEWS",
      "reason": "Google (Alphabet) in the news: \"Alphabet offloads majority stake in GFiber\".",
      "priority": "MEDIUM",
      "signalId": "sig-0200",
      "signalContent": "Alphabet offloads majority stake in GFiber...",
      "signalUrl": "https://..."
    }
  ]
}
```

## 1. STALE_CONTACT — "Reconnect"

**Status:** Implemented

**Trigger:** Contact not interacted with beyond their tier/custom threshold.

**Insight context:**

- Days since last interaction and threshold (e.g., "No interaction in 43 days (threshold: 30 days).")
- Priority mapped from contact tier: CRITICAL -> URGENT, HIGH -> HIGH, MEDIUM/LOW -> MEDIUM

**Actions:**

- **Primary CTA: "Draft Check-in"** — Opens `DraftEmailPanel` with staleness context passed to `generateEmail()`
- Snooze / Done

**Email draft context:** Partner name, contact name/title/company, days since last contact, last interaction summary, relationship history. Type-specific prompt: "This is a check-in email to reconnect with a contact you haven't spoken to in a while."

## 2. JOB_CHANGE — "Executive Transition"

**Status:** Implemented

**Trigger:** ExternalSignal with type `JOB_CHANGE` within 30 days.

**Label:** "Executive Transition" (renamed from "Job Change" for clarity)

This nudge covers two distinct sub-scenarios, distinguished by whether the signal's `contactId` matches the current contact:

### 2a. Own Role Change — Contact themselves changed roles

The signal's `contactId` matches the contact being nudged.

**Insight context:**

- Role change details from signal content (e.g., "Priya Sharma promoted to Head of Strategic Partnerships at Microsoft")
- Signal source URL (View on LinkedIn)
- Other McK partners who also know this person (via `getPartnerRelationsForPerson()`, excluding current partner)

**Actions:**

- **Primary CTA: "Draft Congratulations"** — Opens `DraftEmailPanel` with congratulations context
- Snooze / Done

### 2b. Company Executive Transition — Someone else at the company changed roles

The signal's `contactId` does NOT match the current contact (it's a company-level signal).

**Insight context:**

- Transition details framed as company event: "Executive transition at Google (Alphabet): Brian Foster announced new role: CTO"
- Signal source URL
- **McK partner who owns the relationship** with the person who changed jobs — looked up via `prisma.contact.findUnique()` on the signal's `contactId`, returning the partner name

**Actions:**

- **Primary CTA: "Draft Congratulations"** — Opens `DraftEmailPanel` with executive transition context
- Snooze / Done

**Partner relationship rendering:** The `InsightItem` component renders a purple-themed line: "Sam Rivera knows Brian Foster" with partner names, enabling the current partner to coordinate outreach.

**Implementation:**

- [src/lib/services/nudge-engine.ts](src/lib/services/nudge-engine.ts) — `getPartnerRelationsForPerson()` queries contacts by name + companyId
- Metadata stores `relatedPartners[]` and `personName` per insight
- [src/app/nudges/page.tsx](src/app/nudges/page.tsx) — `InsightItem` renders partner context inline

## 5. MEETING_PREP — "Meeting Prep"

**Status:** Implemented

**Trigger:** Meeting within 0-3 days where the contact is an attendee.

**Insight context:**

- Meeting title (e.g., "Technical Deep Dive: Enterprise Architecture with Google (Alphabet)")

**Actions:**

- **Primary CTA: "Generate Brief"** — Opens `MeetingBriefPanel` (NOT the email draft panel)
- **Secondary: "Draft Email"** — Available as a secondary button when meeting prep is the primary insight
- Snooze / Done

**Meeting brief generation:**

- Endpoint: `POST /api/nudges/[id]/meeting-brief` ([src/app/api/nudges/[id]/meeting-brief/route.ts](src/app/api/nudges/[id]/meeting-brief/route.ts))
- Extracts meeting title from nudge reason, looks up the meeting record with all attendees
- Gathers each attendee's recent interactions and signals
- Calls `generateMeetingBrief()` from [src/lib/services/llm-service.ts](src/lib/services/llm-service.ts)
- Returns structured brief with sections: Meeting Context, Attendee Insights, Recommended Agenda, Suggested Questions, Risks and Watch-outs, Preparation Checklist
- Caches result in `meeting.generatedBrief`
- `MeetingBriefPanel` renders the brief as formatted markdown with interactive checkboxes, copy, and regenerate actions

## 9. LINKEDIN_ACTIVITY — "LinkedIn Activity"

**Status:** Implemented

**Trigger:** ExternalSignal with type `LINKEDIN_ACTIVITY` within 14 days, where `contactId` matches the current contact (not company-wide). Capped at 2 per contact.

**Insight context:**

- LinkedIn activity summary (truncated to 150 chars, e.g., "Active on LinkedIn: 'Brian Foster shared thoughts on agentic AI'")
- Signal source URL with "View on LinkedIn" link

**Actions:**

- **Primary CTA: "Draft LinkedIn Email"** — Opens `DraftEmailPanel` with LinkedIn activity context
- Snooze / Done

**Coherence rule:** LinkedIn activity is only attributed to the actual person who posted/shared/commented. The engine filters by `s.contactId === contact.id` to prevent company-wide LinkedIn signals from being misattributed to other contacts at the same company.

**Email draft context:** The specific LinkedIn activity, how it connects to the partner's work or shared interests, previous interactions. Type-specific prompt: "This is an outreach email inspired by the contact's recent LinkedIn activity."

**Schema:** `linkedinActivityEnabled Boolean @default(true)` on `NudgeRuleConfig`
**Settings UI:** "LinkedIn Activity" toggle on the Nudge Preferences page ([src/app/nudges/settings/page.tsx](src/app/nudges/settings/page.tsx))

## Implementation Architecture (Completed)

### Schema Changes

- **Nudge model** ([prisma/schema.prisma](prisma/schema.prisma)):
  - `metadata String? @map("metadata")` — JSON field storing consolidated insights array
  - `generatedEmail String? @map("generated_email")` — Cached LLM-generated email draft
- **NudgeRuleConfig model**:
  - `linkedinActivityEnabled Boolean @default(true) @map("linkedin_activity_enabled")`

### API Endpoints

- `**POST /api/nudges/[id]/draft-email`** ([src/app/api/nudges/[id]/draft-email/route.ts](src/app/api/nudges/[id]/draft-email/route.ts))
  - Generates type-specific email draft using `generateEmail()` with `RULE_TYPE_CONTEXT` prompts
  - Fetches nudge with contact, partner, company, and signal relations
  - Gathers recent interactions and signals for context
  - Caches result in `generatedEmail` (non-blocking — cache write failure does not fail the request)
  - Returns `{ subject, body }`
- `**POST /api/nudges/[id]/meeting-brief`** ([src/app/api/nudges/[id]/meeting-brief/route.ts](src/app/api/nudges/[id]/meeting-brief/route.ts))
  - Extracts meeting title from nudge reason, looks up meeting record with attendees
  - Gathers each attendee's interactions and signals
  - Calls `generateMeetingBrief()` for structured brief
  - Caches result in `meeting.generatedBrief`
  - Returns `{ meetingTitle, meetingTime, attendeeCount, brief }`
- `**GET /api/nudges`** — Returns nudges with `metadata` field included (via Prisma `findMany` include)
- `**PATCH /api/nudge-rules`** — Supports `linkedinActivityEnabled` toggle

### Nudge Engine ([src/lib/services/nudge-engine.ts](src/lib/services/nudge-engine.ts))

The engine follows a **collect-then-consolidate** pattern:

1. **Collect**: For each contact, gather all applicable insights across all 9 rule types
2. **Cap**: Limit company news to 3 and LinkedIn activity to 2 per contact
3. **Consolidate**: Pick the highest-priority insight as the primary `ruleType`, build a summary reason, and store all insights in `metadata`
4. **Write**: Delete old OPEN nudges, create one new nudge per contact

Key functions:

- `pickPrimary(insights)` — Sorts by priority rank then type rank, returns the top insight's type/priority/signalId
- `buildReason(contactName, companyName, insights)` — Generates "N reasons to reach out..." summary
- `getPartnerRelationsForPerson(name, companyId, excludePartnerId)` — Finds McK partners who know a specific person

### UI Components ([src/app/nudges/page.tsx](src/app/nudges/page.tsx))

- `**InsightItem`** — Renders a single insight with type-specific icon, color, reason text, signal link, and partner relationship context
- `**NudgeCard`** — Consolidated card with contact header, priority badge, reason summary, insights list in a bordered panel, primary CTA, and Snooze/Done actions
- `**DraftEmailPanel`** — In-app email draft with editable subject/body, "Open in Email" (mailto), "Copy", and "Regenerate" actions
- `**MeetingBriefPanel`** — Structured brief renderer with formatted markdown, interactive checkboxes, copy, and regenerate actions
- **Type filter** — Checks `metadata.insights` types (not just `ruleType`) so filtering by "Company News" shows all contacts that have at least one news insight

### Dashboard ([src/app/dashboard/page.tsx](src/app/dashboard/page.tsx))

- Top 5 nudges displayed with consolidated design
- Compact insight list (up to 3 insights shown, "+N more" for overflow)
- No type badges next to contact name (kept concise)
- "Take action" link to nudges page

### Push Email Digest ([src/lib/services/email-service.ts](src/lib/services/email-service.ts))

- Type-specific icons, colors, and CTA labels for each nudge in the digest
- Helper functions: `ruleTypeLabel()`, `ruleTypeIcon()`, `ruleTypeColor()`, `ruleTypeCta()`
- Type breakdown section showing counts per nudge type
- "Manage your nudge preferences" link in footer

### LLM Service ([src/lib/services/llm-service.ts](src/lib/services/llm-service.ts))

- `generateEmail()` — Accepts `nudgeReason` with type-specific context prefix from `RULE_TYPE_CONTEXT` map
- `generateMeetingBrief()` — Generates structured brief with Meeting Context, Attendee Insights, Recommended Agenda, Suggested Questions, Risks, Preparation Checklist
- Template fallbacks for both functions when OpenAI is unavailable
